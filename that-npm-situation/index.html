That NPM Situation

Posted on 23 March 2016

Comments

For a few hours overnight, there was utter panic in the Node.js community. left-pad, a package that consists of an eleven-line JS file, was unpublished from the Node Package Manager registry. You may think that this is a minor thing, but certain packages depended on left-pad. And certain, very popular packages depended on those. The result was broken builds, broken builds everywhere.

So this actually brings up a whole bunch of big questions for the JavaScript community. The first of those being, why did the maintainer remove all of their packages from NPM? The chap in question, Azer KoÃ§ulu, wrote a very interesting and thought provoking Medium article, stating that he believes that "NPM is someoneâs private land where corporate is more powerful than the people" due to actions over a copyright claim. He also goes on to say that his code is still available on GitHub and he believes in open source. So really, although he did break a lot of people's projects, you gotta respect his integrity, right? It's something he did in the name of Open Source, and you can't really argue against that (although, I imagine that it wasn't the easiest decision to make).

However, it brings up yet more moral dilemmas for awful tech commentators (both casual, like me, and professional, like not-me). Hours after left-pad was removed, the CTO of NPM, Laurie Voss, stepped in and added left-pad back to the registry himself, in what he described as an 'unprecedented' action.

Hey npm users: left-pad 0.0.3 was unpublished, breaking LOTS of builds. To fix, we are un-un-publishing it at the request of the new owner.

— Laurie Voss (@seldo) March 22, 2016

And again, you may count it as a 'wrong' action, but it's deciding between whether to support millions of projects or support one guy. And in this case they went with the former. And in NPM's defence again, left-pad was released under the WTFPL, which reads as:

DO WHAT THE F**K YOU WANT TO PUBLIC LICENSE Version 2, December 2004 Copyright (C) 2004 Sam Hocevar sam@hocevar.net

Everyone is permitted to copy and distribute verbatim or modified copies of this license document, and changing it is allowed as long as the name is changed.

DO WHAT THE F**K YOU WANT TO PUBLIC LICENSE TERMS AND CONDITIONS FOR COPYING, DISTRIBUTION AND MODIFICATION

  1. You just DO WHAT THE F**K YOU WANT TO.

So, I guess they can kinda 'do what the f**k' they want to.

...the rest of the dominos fall like a house of cards. Checkmate.

Another big question we should be looking at is, why should a user removing one package containing eleven lines of code break thousands of projects, builds and other packages? It exploits a huge flaw with NPM that should ideally be addressed. Perhaps rather than code being deleted, it is 'archived' and cannot be viewed on the registry any more, while still being able to be accessed by packages that use it already. Or perhaps the entry should be orphaned, for someone else to take on; since packages have open source licenses, it should easily allow other maintainers to take over.

Or perhaps, the whole model is wrong. Look at Linux distributions; their package managers tend to let you add/remove repositories, or servers where be packages. For instance, Arch Linux has core, extra and community repositories amongst others, but there is no stopping me, a user, from adding the custom infinality repository, which isn't owned by Arch Linux but provides binary files of the infinality font program. As far as I know this is not possible with npm, the command line tool for the registry; that you can only choose one registry in the config file. It seems like you need to depend on extra tools to get the ability to use many registries at the same time, for instance StrongLoop. Or at least it's currently something set in the package.json file, something discussed in NPM issue #1401.

To conclude, there are some very pertinent issues with Node.js and NPM right now, but it seems like that until a truly open source competitor is created (and is adopted to the same level), we need to make steps to make do with what we've got.
